Go 版本为 1.25.7，实际开发中应该使用 Go 1.22 及以上的现代语法规范，而避免局限于老旧写法、库等。

作为一个学习类的项目，我们默认这个项目编译运行于 Ubuntu 22.04 / Debian 12 或者更新的操作系统上，不必担心兼容性一类的问题。

本机安装了 golangci-lint，请使用它多进行格式检查和规范等内容。

如果需要报错打日志啥的，记得用 log/slog，而不是单纯的打到 stdErr 里。

测试应该基于表格驱动测试。

比起大量使用 `if got!= want`，我个人建议使用 testify 里的断言进行替代。

## 编码风格：语义优先

项目作者是退役算法竞赛选手，Go 经验尚浅、没有历史包袱。因此代码风格遵循以下原则：

### 核心理念

**让代码"读起来就是它做的事"**——优先选择语义最直白的写法，而不是仅仅追求"lint 通过"。对于一个 Go 新手来说，读完一行代码后不应该需要额外的心智负担去理解"它实际上在干什么"。

### 错误处理

遵循 Go 的标准错误处理三件套，根据**语义**选择对应的方式：

| 场景 | 写法 | 示例 |
|------|------|------|
| 静态错误消息，无需格式化 | `errors.New(...)` | `errors.New("missing required flag: --exec")` |
| 需要携带动态数据 | `fmt.Errorf("...%d...", val)` | `fmt.Errorf("unsupported language: %q", raw)` |
| 为已有 error 附加上下文 | `fmt.Errorf("...%w", err)` | `fmt.Errorf("failed to parse flags: %w", err)` |

**不要**用 `fmt.Errorf("%s", msg)` 来代替 `errors.New(msg)`——前者在语义上是"格式化一个字符串为错误"，后者直接说的是"创建一个新错误"，意图更清晰。

### 命名

- 避免遮蔽内置函数（`min`、`max`、`cap`、`len`、`new` 等），哪怕编译器允许。用更具领域含义的名字替代（如 `limit` 代替 `cap`）。
- 在**存在歧义或角色不明显**的地方，用稍具体的名字点睛。但不要过度——Go 惯用的短命名（`i`、`n`、`ctx`、`err`、`ok`、`buf` 等）在作用域小、含义显然的场合完全没问题，不必刻意展开为 Java 风格的长名。

### 现代化但不求新

大致采用 Go 1.22+ 的**主流规范写法**（`for range N`、`math/rand/v2`、`slices`、内置 `min`/`max` 等），但不追逐实验性特性。以"大多数 Go 项目已经在用"为标准。
