# 产品需求文档 (PRD)：算法竞赛评测引擎

## 1. 产品主旨

构建一个优雅、简洁、高效的算法竞赛评测引擎（Judge Engine），系统底层基于 Go 语言实现，旨在提供高性能、高隔离度、高扩展性的代码评测服务。

## 2. 系统架构设计

系统采用分层架构设计，自上而下将请求解析、业务调度、任务抽象、容器调度与数据存储进行解耦，共划分为六个核心层级。希望遵循以下原则：

1. Functional Core, Imperative Shell（纯函数核心，命令式外壳）
3. Hexagonal Architecture（六边形架构 / 端口与适配器）

## 3. 核心模块与功能说明

### 3.1 Web 处理层

对外提供统一的 HTTP 接口，负责接收外部的评测请求，解析并校验参数结构。单次评测请求必须包含以下核心参数：

| **字段名称** | **类型** | **描述说明**                                                 |
| ------------ | -------- | ------------------------------------------------------------ |
| 源代码       | 字符串   | 选手提交的代码纯文本内容                                     |
| 语言类型     | 字符串   | 目标编译与运行语言标识                                       |
| 限制条件     | 对象     | 包含运行时间限制 (Time Limit) 和空间限制 (Space Limit)       |
| Checker      | 字符串   | 校验规则定义。可填 `default`（忽略行末空格与回车）、或 Testlib 标识（如 `ncmp`, `rcmp6`，无后缀扩展名），或指定外部存储池中的文件名作为特判程序（SPJ） |
| 测试数据     | 数组     | 若干组测试用例。单组数据格式支持纯文本直接传输，或传入指向外部存储池的文件路径引用 |

### 3.2 业务逻辑层 1（流程调度）

作为单次评测的主控链路，负责编排完整的判题生命周期。其标准工作流如下：

1. 发起选手源代码的编译请求。
2. 同步准备该题目所需的测试数据。
3. 识别并编译本次评测所需的 Checker 程序。
4. 遍历所有测试数据组：执行选手程序 -> 若沙盒容器正常结束且未超时越界，则拉起 Checker 对照标准输出进行校验，判定是否 AC (Accepted)。

### 3.3 业务逻辑层 2（任务封装）

对底层抽象的执行动作进行高阶封装，向流程调度层提供四个标准化的原子业务接口：

- **编译源代码**：封装特定语言的编译参数与指令。
- **编译 Checker**：封装校验程序的编译过程，默认深度集成 Testlib 框架环境。
- **运行选手程序**：封装资源限制注入与标准输入输出重定向逻辑。
- **运行 Checker**：封装校验环境的启动参数与判定逻辑。

### 3.4 编排层

系统任务的调度总线。负责接收来自业务层的所有“容器调用”请求指令（无论该请求是用于编译还是运行）。编排层负责平滑对接底层沙盒集群，进行任务分发与状态监听。

### 3.5 容器层 (Sandbox)

纯粹且隔离的底层沙盒环境。不感知任何上层业务逻辑（例如“这是否是一个 Checker”），仅根据传入的指令集、限制参数及挂载的文件，在安全受限的环境内负责具体的“编译”或“执行”动作，并回传进程退出码、标准流与资源消耗明细。

### 3.6 存储与资源层

统一管理评测过程中的所有静态资产与动态产物，依据读写特性划分为三个核心池：

| **存储模块**   | **核心职责与特性**                                           |
| -------------- | ------------------------------------------------------------ |
| **缓冲池**     | 内置 LRU 淘汰策略，用于临时缓冲编译产出的二进制文件（选手可执行文件、Checker）及从外部拉取的测试数据文件。**注**：Web 层随 HTTP 载荷直接传入的纯文本测试数据无需也暂不进入此缓存。 |
| **内部存储池** | 静态资产库。用于存放系统默认运行必须依赖的文件，例如系统自带的标准 Testlib 头文件 (`testlib.h`) 及各类预置标准 Checker 源码等。 |
| **外部存储池** | 外部资源挂载点。对接外部存储设施，负责存放从其他服务或业务线拉取的外部文件，如大体积评测数据、特定依赖的二进制文件或自定义的 SPJ 源码等。 |
